What this is

Program Command (a.k.a. EWU Design Schedule Analyzer) is the scheduling, workload, and capacity-planning surface for the EWU Design program. Vanilla JS frontend, Supabase (Postgres) for persistence, Chart.js for visualization. Production is a static GitHub Pages site that reads a public Supabase project; editing requires sign-in plus membership in a server-side allowlist.

One maintainer (intermittent) supervises autonomous agents that generate branches and PRs. The operating model — trunk-based, manual deploy, RLS as the data-safety control, agents on a forbidden-path leash — is shaped by that constraint, not by abstract preference.

Get the app running locally

# Serve the static site (no build step)
python3 -m http.server 8080
# or: npm run serve

# Optional: run the Node API server (Sheets export endpoint)
node api-server.js          # port 5000

# Tests
npm test                    # Jest + jsdom
npm run test:watch
npm run test:coverage

Tests live under tests/ and match **/*.test.js. Coverage collects from js/**/*.js excluding js/config/.

Architecture at a glance

Frontend layers

  • Pages (pages/*.html + pages/*.js) — each dashboard pairs an HTML file with a controller JS file. public-schedule.html lives at the repo root because it is the public surface; since PR #256 the site root (index.html) redirects to it, and the editing app lives at program-command.html.
  • Shared modules (js/) — domain singletons in the IIFE pattern (ScheduleManager, FacultyManager, ReleaseTimeManager, StateManager, etc.). Loaded as global <script> tags; do not convert to ES modules.
  • StateManager (js/state-manager.js) — central, subscription-based, with localStorage persistence.
  • Engines — ConflictEngine and ConstraintsEngine for schedule validation.
  • Data access — js/db-service.js wraps Supabase reads/writes with automatic fallback to static JSON when Supabase is unreachable.

Persistence

  • Supabase — primary store for courses, faculty, constraints, release-time allocations.
  • localStorage — schedule edits, placements, backup snapshots. Prefix ewu_schedule_.
  • Static JSON — fallback data (workload-data.json, enrollment-dashboard-data.json, data/*.json).

Node surface

api-server.js serves static files on port 5000 and exposes one endpoint, POST /api/export-to-sheets, backed by google-sheets-client.js. It does not run on GitHub Pages, so the Sheets export only works locally (npm run serve). The export is a real, needed function for faculty workloads; the decision (2026-05-21) is to keep it and rehost it for the static-Pages surface — see the decision log. Do not delete it.

Forbidden path

js/supabase-config.js is on the agent forbidden-path list. Project routing is a safety control; only humans edit it.

Row-level security and the editor allowlist

The prod Supabase anon key is public by design. RLS is therefore the real data-safety control:

  • Reads — "Public read" policy: SELECT to public USING true on the 9 operational tables. Anon and authenticated can read freely.
  • Writes — editor_insert/update/delete_* policies require public.is_editor(), which checks membership in the public.editors table.
  • Allowlist — 4 rows in public.editors: tmasingale@ewu.edu, tmasingale@me.com, mbreen@ewu.edu, rhunton1@ewu.edu. Mutable only via direct SQL on the prod DB.

The public-schedule path additionally goes through a SECURITY DEFINER RPC, so it would survive a future tightening of anon SELECT. Tightening anon-read is explicitly out of Phase 0 scope (see the Phase 0 runbook appendix).

Deploy flow

The model

agents ─┐
you ────┴─▶ short-lived branch ─▶ PR ─▶ [CI: test + onboarding required] ─▶ main
                                                                            │
                                                                            ▼
                                            YOU run "Deploy to Production"
                                            (workflow_dispatch, per-SHA CI-green gate)
                                                                            │
                                                                            ▼
                                            GitHub Pages ─▶ prod Supabase

What this means in practice

  • Nothing reaches production on a push to main. The deploy workflow's only trigger is manual workflow_dispatch.
  • The deploy job's first step queries the GitHub API for the deployed commit's test and onboarding check-runs and aborts unless both are success. The gate is keyed to the exact SHA being deployed — a later un-CI'd commit cannot ride a stale green.
  • After publishing, the deploy job stamps docs/ORIENT.html's prod-state-as-of marker. ORIENT's freshness is mechanical, not by convention.

Branch protection on main

Enforced via the protect-main repository ruleset (not classic branch protection — the classic branches/main/protection API returns 404 by design):

  • Required status checks: test and onboarding, strict (branch must be up to date). The onboarding check is static-artifact validation — never a live /onboarding skill invocation, so the maintainer can never be locked out by external skill availability.
  • Pull request required before merge; 1 approving review required. Direct push, force-push, and branch deletion are all rejected.
  • The repo-admin role is the sole bypass actor — the solo maintainer can merge their own hotfix (GitHub blocks self-approval); agents are non-admin and stay fully bound.
  • Merged head branches auto-delete.

Autopilot rules and forbidden paths

Agents are expected to:

  • Target main only. No develop work.
  • Use short-lived branches; one feature per PR.
  • Never trigger a production deploy.
  • Never edit any of the forbidden paths below.

Forbidden paths (enforced since 2026-05-21 by the required guard-forbidden-paths check — .github/workflows/guard-forbidden-paths.yml, shipped in PR #255 — in addition to the policy in AGENTS.md):

  • .github/workflows/** — modification and creation. Closes the "add a new auto-deploy workflow" bypass.
  • ci.yml
  • js/supabase-config.js
  • .env*
  • Supabase policy / migration SQL files

Override

The only override is the human-applied allow-forbidden-paths PR label, which an agent cannot self-apply. Authorship is checked per commit, not by PR author, so an agent cannot launder edits through a delegated PR.

Domain conventions

Workload multipliers

  • Regular courses: credits × 1.0
  • DESN 495 (Internship): credits × 0.1
  • DESN 499 (Independent Study): credits × 0.2
  • DESN 491 (Practicum): credits × 0.15

Faculty annual credit limits

  • Professors (Full / Associate / Assistant): 36
  • Lecturers (Senior / Regular): 45
  • Adjunct: 15

Utilization thresholds

  • over > 100%
  • optimal 60–100%
  • under < 60%

Data formats

  • Course codes: DESN XXX (e.g. DESN 499)
  • Academic years: YYYY-YY (e.g. 2024-25)
  • Quarters: Fall, Winter, Spring, Summer
  • Faculty configuration: scripts/faculty-mapping.json
