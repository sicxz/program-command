Must-preserve behavior checklist

This is the Unit 10 verification manifest: each item needs an explicit post-resolution check on main — npm test alone does not satisfy the gate. It doubles as the develop-deletion checkpoint's test manifest (Unit 11 precondition).

Develop-only behaviors to land

  • db-service tenant-aware initialize(): re-init on active department change, departmentIdByCode UUID cache, dynamic departments-row get-or-create from getActiveDepartmentIdentity().
  • supabase-config exposes getActiveDepartmentProfile() / getActiveDepartmentIdentity() for shared modules.
  • BackupManager.init({storageNamespace}) and ReleaseTimeManager.init({storageNamespace}): sanitized per-profile prefixes, no cross-program leakage, category relabeling from the active program's course codes.
  • COURSE_CODE_PATTERN generalized to /^[A-Z]{2,6}\s\d{3}[A-Z]?$/ in both constants.js and validators.js.
  • Release-time dashboard fully profile-driven (years, quarters, roster, titles — no hardcoded 8-name Design list).
  • Schedule builder: getScopedStorageKey at all 8 localStorage sites (scoped-first read, legacy fallback, scoped-only write); non-DESN empty-faculty gating + renderer early-return; applyDepartmentProfileCopy; populateAcademicYearControls.
  • program-command.html: profile-scoped displacedCourses/customFaculty keys with one-time migrateLegacyProfileScopedStorage, hydration strictly after migration (develop users' data lives only under scoped keys — losing the order orphans it).
  • Shell boot gate: conditional ProgramCommandShell.bootstrap({launchRuntime: init, …}) with catch→toast→init() fallback; bare init() only when the module is absent.
  • Shell catalog model: 15-entry flatten, CSEE type:'department' group, workspaceKind taxonomy, memberProgramIds, identity overrides; programCommandShellSelectionV1 / programCommandOnboardingContextV1 localStorage contracts; data-detection probe order.
  • Batched screenshot intake: webkitdirectory upload with capability detection, MIME/extension filter, year+summer-aware inferScreenshotTermYear, group sort contract.
  • Onboarding handoff: handoff card, identity prefill, grouped batches with inferred-term warnings, persisted catalog metadata.
  • Constraints service (as amended): profile-driven lookup with explicit missing-row throw; zero Design room-restriction fallbacks; empty reachable table ⇒ zero constraints.
  • Applied Learning + workload dashboards + faculty-workload-detail: all profile-driven (courses, multipliers, years, identity, reset year), any-N-courses rendering.
  • Onboarding generator derivations (storage prefix, branding, next-year reset default, course-code remapping) emitting strict-schema-complete profiles.
  • All six rescued develop test files green on main.

Main-side keepers (must survive every graft)

  • updated_by save attribution on all write paths + lastSaveAttribution; pathways API; PKCE auth; auth-guard script tags on every page.
  • Schedule builder Phase A–E (dirty tracking, modal a11y, keyboard drag-drop), draft-first source loading, workload-review handoff, profile-driven TIMES/DAYS, CONFLICTS taxonomy.
  • js/schedule-data-utils.js superset (all 6 exports incl. scheduleHasCourses, buildScheduleDataFromDatabaseRecords) — public landing and CI depend on them.
  • Strict v1 profile schema (0efa865) and its validation test.
  • Admin-vs-chair onboarding gate — changes only via adjudication (a), never as a merge side effect.

Top risks

  • Auth policy collision (G6) — adjudicate (a) before starting G6 or the rescue silently weakens Phase 0/1 auth.
  • One-way storage migration (G4) — the legacy-key delete-after-copy plus hydrate-after-migrate ordering must survive conflict resolution exactly.
  • program-command.html churn — main's fastest-moving file; G4/G6 must rebase fresh, and the boot-gate conditional is the likeliest hunk to be lost to a take-main resolution.
  • RLS interaction — the shell's Supabase head-count probes may return zero/denied for unauthenticated users under the hardened policies, mis-routing populated programs into onboarding; the Unit 10 integration scenario must exercise this.
  • Catalog double-modeling — 27faa5c's "CSEE" department vs PR #227's "EECS" catalog: Unit 12 must reconcile to one model (coordinated-pair handling for #227/#229 is already plan-mandated).
  • department-onboarding.js hand-merge — patched by G5 and twice by G6; the highest-skill merge in the rescue. Keep the G5→G6 order.
