What was applied (Path B — editor allowlist)

During the read-only production characterization (step 1), the original plan's target schema was found not to exist in either Supabase project. Rather than patch around a migration the codebase has never exercised end-to-end, Phase 0 shifted to a smaller, focused change that closes the actual exploitable gap: the existing auth_* write policies allowed any signed-up account to write to every operational table.

Path B replaces the write policies with an explicit editor allowlist while leaving the existing "Public read" SELECT policies untouched, so the public schedule and the editor's read paths continue to work.

What is intentionally deferred

  • The program-scoped tenant migration (t02 → t04 → t03; PR #246). The SQL assumes tables (pathways, pathway_courses, programs, user_programs) that don't exist in production yet, and has never been exercised against a real Supabase project (CI uses mocks). Re-scoping this is its own effort: stub or drop the missing tables from the migration arrays, exercise the apply path end-to-end against a real test database, then map the existing auth.users accounts to user_programs rows before enabling tenant-scoped RLS — otherwise the editor app silently returns zero rows for users without a program claim.
  • Tightening "Public read": anon currently reads all 9 operational tables directly. The public schedule path goes through a SECURITY DEFINER RPC, so anon-table-read can be revoked if/when nothing else relies on it. Out of scope for Phase 0.
Appendix — Original Path A plan and why it was abandoned (audit trail)

This is preserved verbatim so the decision history remains visible. The plan below was the runbook's content at the time PR #251 was opened. Characterization disproved its premises; Path B above is what actually shipped.

What was planned (and why it didn't run)

  1. Snapshot prod public.programs policies. Why it didn't run: public.programs does not exist in production. The migration was to create it.
  2. Merge PR #246. Why it didn't run: #246 only adds new files (t04 SQL, smoke check, docs, tests). The actual RLS policy SQL (t03) was already on main and not part of #246's diff, so merging alone changes nothing in prod. Risk lived entirely in the manual SQL apply step, which was found to be unrunnable as written.
  3. Apply t02 → t04 → t03 against the production Supabase project. Why it didn't run: t02 tries to ALTER TABLE pathways and pathway_courses — neither table exists in production. The entire transaction would fail on those statements. The develop project (cstcwplvioheazoghkgf) is in the same state (contains only an orphan public.programs), so no real environment had ever exercised this SQL.
  4. Run the smoke check against production. Why it wasn't sufficient as written: the script's "authorized" client takes SUPABASE_AUTH_KEY, which the runbook suggested could be the service role. Service role bypasses RLS, so the "authorized allowed" assertion would have passed trivially without ever exercising the real authenticated path. The Path B verification (scripts/phase0-policy-verify.mjs) replaces it with a flow that signs in a real editor JWT and tests a throwaway non-editor user from the admin API.

The real exploitable gap that was actually present

Production RLS was already on for the 9 operational tables under an undocumented scheme:

  • "Public read" SELECT TO public USING true — anon and everyone read freely.
  • auth_insert/update/delete_* policies with auth.uid() IS NOT NULL — any signed-in account could write anything.

Combined with signUp being enabled in js/auth-service.js, the actual risk was "any stranger who registers an account can write to any operational table," not "no RLS." That is the gap Path B closes.

Lesson for future runbooks

Before any "apply this migration to prod" runbook is approved, run a read-only characterize step against the target. SQL files that pass CI mocks have not necessarily ever been executed against a real database, and the actual prod state is often different from the prose description.
